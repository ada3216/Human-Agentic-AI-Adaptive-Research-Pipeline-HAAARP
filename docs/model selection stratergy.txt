The unusual property of this devplan
Most software devplans under-specify. This one over-specifies — every function has exact signatures, return types, error codes with shell exit values, and named acceptance criteria. That's actually GPT-4.1's sweet spot. Where models typically hallucinate architecture decisions, this plan makes them unnecessary. The model's job is closer to "transcribe this specification into working Python" than "design and build a system."
This changes the capability ceiling calculation significantly.

What free tier can handle phase by phase
Phase 0 and 1: entirely on free tier, GPT-5 mini can do most of it. This is folder creation, YAML config, Makefile, markdown docs, adapter scaffolds with # TODO markers. No reasoning complexity at all. One session, GPT-4.1 as executor, done.
Phase 2a (ingest, pass1, lens dialogue): GPT-4.1 handles this well. ingest_and_deid.py and pass1_runner.py are both well-specified file-writing + Ollama REST calls. lens_dialogue.py is slightly more complex (10-question structured interview, lock mechanism, ORCID validation) but the spec is exact enough that it's implementation, not design. One thing the 85% framework catches here that bare Copilot might not: the GuardrailSpec forces explicit pseudocode for lock_lens() before implementation, which surfaces the question of what happens if docs/lens.md doesn't exist yet when the agent tries to extract the questions. That's a real sequencing ambiguity the framework catches at plan time.
Phase 2b (pass2_runner, stability loop, grounding_checker, audit_emitter): this is where it gets genuinely complex. pass2_runner.py has six sequential pre-condition checks with specific error codes and exit values — GPT-4.1 can implement all six correctly but this is the first place you want o4-mini doing the phase gate review, not GPT-5 mini. The stability loop (four runs, different seeds, Jaccard computation across all pairs) is implementable but needs careful review. grounding_checker.py is the most complex single function — cross-referencing claims against transcript locations, computing support_strength from excerpt counts and participant counts, flagging multiple hallucination types. GPT-4.1 will produce something plausible but the correctness of the cross-reference logic specifically needs human verification. audit_emitter.py is actually simpler than it looks — it's mostly validation checks and JSON assembly.
One genuine gap in the devplan: lens_amplification_index is defined as a field in stability_report but never explained. What's the formula? No framework — free tier, paid tier, 85% or otherwise — will make a model compute something that isn't specified. This is a human-authored gap, not a model capability gap. Clarify it before phase 2b starts.
Phase 3 (14 named tests, CI, demo notebook): GPT-4.1 handles test-writing for well-specified behaviour very well, especially when the test names are already given and the acceptance criteria are exact. The MOCK_LLM=true mock pattern is standard. This phase is actually safer than 2b.
Phases 4 and 5 (release docs, ethics docs, handover checklist): pure markdown, any model, any tier.
Phase 6: straightforward optional extensions, implement if and when needed.

Does the 85% framework extend capability or just reduce errors at the same ceiling?
Both, but in different ways for this specific project.
It genuinely extends capability in one important way: the sequencing enforcement. This project has hard artifact dependencies — pass2 cannot run without pass1 anchor deposited to OSF, audit emitter refuses to package if anchor_type is local. These are domain-level sequencing requirements, not just code dependencies. The 85% framework's state machine and risk gates align naturally with this. The dependency declarations in Addition 4 map directly onto the pass1 → OSF deposit → pass2 chain. Without the framework, a Copilot session might start implementing pass2 runner while mentally assuming the anchor will exist — with the framework, the plan explicitly declares depends-on: pass1_anchor_with_osf_doi and that dependency is checked before execution.
It reduces errors at the same ceiling in the usual way: GuardrailSpec catches the lens_amplification_index ambiguity at planning time rather than letting GPT-4.1 invent a formula. Risk gates fire on pass2_runner.py (six-condition gate logic qualifies as auth/security equivalent) and on audit_emitter.py (packaging logic with blocking conditions). The scope classification correctly identifies phases 2a and 2b as STRUCTURAL — they touch multiple files, exceed 300 LOC, and introduce new public interfaces — which triggers human confirmation before the executor starts.
What it doesn't change: if GPT-4.1's grounding_checker.py cross-reference logic is subtly wrong — tracking location references correctly, handling missing segments, correctly counting distinct participants — the framework will catch that the tests fail, not that the logic was wrong before testing. You still need to read the grounding checker output carefully yourself.

On local model parity with free tier:
Roughly yes, with important nuance. Qwen3.5-27B is around GPT-4.1 level on structured coding tasks when the spec is explicit — as it is in this devplan. GLM-4.7-Flash is similarly competitive for review/verification passes with its consistency advantage. o4-mini is ahead of the local 27B models specifically on multi-step reasoning chains and constraint satisfaction — the kind of logic in pass2_runner.py's six-condition gate. The 122B-A10B MoE narrows that gap but introduces the cache-thrashing issue in agentic contexts.
The "play to their strengths" framing is right. Where local models hold up well against free tier: file-bounded implementation tasks with explicit specs, test writing, documentation. Where they fall behind o4-mini specifically: reasoning through implicit constraint interactions, catching contradictions across a long spec, multi-file dependency analysis. For this devplan specifically, given how explicit the spec is, Qwen3.5-27B could handle phases 0, 1, 3, 4, 5 at roughly the same quality as GPT-4.1.
